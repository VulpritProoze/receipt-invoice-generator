# Design Auditor

Crawls a client-side codebase within a given scope or folder and produces a `DESIGN.md`
— a structured, human- and agent-readable document that captures the design language,
component conventions, aesthetic style, and visual rules as they actually exist in code.

The output is a living reference doc: new developers and AI agents can read it before
creating anything new to stay consistent with the existing design system.

---

## When to Use

- User says: "generate a DESIGN.md", "document the UI style", "analyze the frontend design"
- A new feature needs new components and the agent needs design context first
- An AI agent is about to build UI and should match existing visual conventions
- The team has no formal design system doc but has an established visual codebase

---

## Instructions

### Step 1 — Confirm Scope

Before scanning, confirm:
1. **Root folder** — which directory to analyze (e.g. `client-side-ts/src`, `src/components`)
2. **Depth** — full audit (everything) or focused (components only, styles only, etc.)
3. **Framework hint** — if not obvious from files (React, Vue, Svelte, Angular, plain HTML)

If the user already provided a path or scope, proceed without asking. State your assumptions.

---

### Step 2 — Crawl and Collect

Systematically read the codebase within the scope. Collect evidence across these layers:

#### A. Project Structure
```bash
find <scope> -type f | sort
```
Map the folder layout. Identify:
- Component directories vs. page directories vs. layout directories
- Style entry points (`index.css`, `globals.css`, `tailwind.config.*`, `theme.*`)
- Token files (`tokens.ts`, `variables.css`, `design-tokens.*`, `constants/colors.*`)
- Asset directories (fonts, icons, images)

#### B. Design Tokens & Variables
Look for and extract:
- **Colors**: CSS custom properties (`--color-*`), Tailwind config `colors`, JS/TS color maps
- **Typography**: font families, sizes, weights, line heights, letter spacing
- **Spacing**: scale values, padding/margin patterns, gap usage
- **Borders & Radius**: border widths, `border-radius` values
- **Shadows**: box-shadow and drop-shadow definitions
- **Z-index**: layering scale
- **Breakpoints**: responsive breakpoint definitions

#### C. Component Patterns
Sample 5–10 representative components. For each, note:
- Structural pattern (compound component, atomic, slot-based, etc.)
- Class/style approach (CSS modules, Tailwind, styled-components, plain CSS, etc.)
- State variants (hover, active, disabled, loading, error)
- Props interface if TypeScript (look for `interface Props` or `type Props`)

#### D. Aesthetic Style
Derive from visual evidence in code:
- **Color palette**: dominant, accent, neutral, semantic (success/warning/error/info)
- **Typography hierarchy**: heading scale, body text, labels, captions, code
- **Layout approach**: grid vs. flexbox, column widths, container max-width
- **Motion & Animation**: transition durations, easing functions, keyframe animations
- **Density**: compact vs. spacious (derive from padding/gap values)
- **Tone**: minimal, playful, corporate, editorial, utilitarian, etc. — infer from choices

#### E. Decorative Conventions
Note recurring visual patterns:
- Icon library in use (Lucide, Heroicons, Font Awesome, custom SVG, etc.)
- Background treatments (gradients, textures, patterns, solid)
- Dividers and separators
- Card/panel styles
- Badge and tag styles
- Interactive feedback (ripples, transitions, skeleton loaders)

#### F. Naming Conventions
- CSS class naming (BEM, utility-first, camelCase, kebab-case)
- Component file naming (PascalCase, kebab-case)
- Token naming patterns (`--color-brand-primary` vs `--primary` vs `colors.brand[500]`)

### Step 4 — Flag Inconsistencies

After writing the doc, add a brief inline note for any inconsistencies found:
- Mixed naming conventions
- Multiple border-radius scales that don't align
- Colors defined in multiple places with different values
- Components that don't follow the dominant pattern

List these under **Section 12 — Open Questions / WIP**.

---

### Step 5 — Confirm and Offer Follow-Up

Tell the user:
- Where `DESIGN.md` was written
- How many components/files were analyzed
- Any major gaps (e.g., "no token file found — colors were inferred from usage")

Offer:
- A focused re-audit on a specific section (colors only, motion only, etc.)
- Generating a `tokens.ts` or `variables.css` if none exists
- A component inventory table as a separate `COMPONENTS.md`

---

## Output Format

- Primary output: `DESIGN.md` written to the analyzed scope folder (or project root)
- Secondary: brief summary in chat of what was found and any notable gaps
- Do NOT output the full DESIGN.md contents in chat — write it to disk and confirm the path

---

## Guardrails

- Do not modify any existing source files — read only within scope
- Do not infer design intent from file names alone; read the actual style definitions
- If a token is defined but never used, note it as "defined but usage unclear"
- If the codebase uses multiple styling approaches (mixed Tailwind + CSS modules), document both — do not pick one as canonical unless one clearly dominates
- Do not hallucinate color values or font names; only document what is present in code
- If scope is ambiguous (e.g., "the frontend"), ask one question to confirm the root path before proceeding
- Keep DESIGN.md factual and descriptive — not prescriptive. This is a record of what exists, not a style guide for what should exist (unless the user asks for that)
