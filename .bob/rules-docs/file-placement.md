# Documentation Mode: File Placement

**Purpose**: Defines where different document types go, naming conventions, and directory organization rules.

---

## Directory Structure

```
docs/
├── architecture/      # System design and module documentation
├── decisions/         # ADRs (Architecture Decision Records)
├── getting-started/   # Setup and usage guides
├── plans/             # Task plans and roadmaps
├── reports/           # Test, refactor, and commit reports
│   ├── commit-reports/
│   ├── refactor-reports/
│   ├── session-logs/
│   │   ├── resolved/
│   │   └── unresolved/
│   └── test-reports/
└── templates/         # Documentation templates
```

---

## Placement Rules by Document Type

### Architecture Documentation

**Target directory:** `docs/architecture/`

**When to use:**
- Documenting system design
- Describing module structure and responsibilities
- Explaining data flow and layer communication
- Mapping dependencies and interfaces

**Naming convention:**
- Use kebab-case
- Be descriptive: `invoice-module.md`, `system-overview.md`, `testing-strategy.md`
- Avoid generic names: not `module1.md` or `notes.md`

**Required metadata:**
- `doc_id`: `ARCH-NNN` (e.g., `ARCH-001`)
- All standard metadata fields

### Decision Records (ADRs)

**Target directory:** `docs/decisions/`

**When to use:**
- Documenting architectural decisions
- Recording technology choices
- Explaining trade-offs and alternatives considered
- Superseding previous decisions

**Naming convention:**
- Format: `DEC-NNN-brief-description.md`
- Example: `DEC-003-testing-stack.md`, `DEC-007-pdf-library.md`
- Use three-digit zero-padded numbers

**Required metadata:**
- `doc_id`: `DEC-NNN` (e.g., `DEC-003`)
- `status`: `proposed`, `accepted`, `deprecated`, or `superseded`
- If superseded: include `superseded_by: DEC-NNN` in metadata

**Special rules:**
- Never delete deprecated ADRs — mark as `deprecated` or `superseded`
- When superseding, update the old ADR to reference the new one
- Maintain chronological numbering (don't reuse numbers)

### Plans

**Target directory:** `docs/plans/`

**When to use:**
- Planning multi-step tasks
- Breaking down complex features
- Documenting roadmaps
- Outlining implementation strategies

**Naming convention:**
- Use kebab-case
- Be specific: `implement-mvp.md`, `local-sqlite-dev-workflow.md`
- Avoid vague names: not `plan.md` or `todo.md`

**Required metadata:**
- `doc_id`: `PLAN-NNN` (e.g., `PLAN-002`)
- All standard metadata fields

### Test Reports

**Target directory:** `docs/reports/test-reports/`

**When to use:**
- After running the test suite
- Documenting test results
- Recording coverage metrics
- Tracking test failures and fixes

**Naming convention:**
- Format: `REP-TEST-NNN-brief-description.md`
- Example: `REP-TEST-001-schema-harness-smoke.md`
- Use three-digit zero-padded numbers

**Required metadata:**
- `doc_id`: `REP-TEST-NNN`
- All standard metadata fields

**Special rules:**
- Always generated by `/run-tests` skill — never hand-written
- Include full test output and coverage metrics
- Document any failures with file and line references

### Refactor Reports

**Target directory:** `docs/reports/refactor-reports/`

**When to use:**
- After completing a refactor
- Documenting structural changes
- Recording file moves, renames, or deletions
- Explaining refactor decisions

**Naming convention:**
- Format: `REP-REFACTOR-NNN-brief-description.md`
- Example: `REP-REFACTOR-001-site-shell.md`
- Use three-digit zero-padded numbers

**Required metadata:**
- `doc_id`: `REP-REFACTOR-NNN`
- All standard metadata fields

**Special rules:**
- Must list every file modified, created, or deleted
- Must explain the reasoning behind the refactor
- Must document any risks or follow-up items

### Commit Reports

**Target directory:** `docs/reports/commit-reports/`

**When to use:**
- Documenting significant commits
- Recording atomic commit details
- Explaining commit scope and impact

**Naming convention:**
- Format: `REP-COMMIT-NNN-brief-description.md`
- Example: `REP-COMMIT-001-atomic-commit-report.md`
- Use three-digit zero-padded numbers

**Required metadata:**
- `doc_id`: `REP-COMMIT-NNN`
- All standard metadata fields

### Session Logs

**Target directory:** `docs/reports/session-logs/unresolved/` (initially)

**When to use:**
- Automatically at session end
- Never manually created

**Naming convention:**
- Format: `SES-[ISO-8601-timestamp].md`
- Example: `SES-2026-05-21T14-00-00.md`
- Timestamp in UTC

**Lifecycle:**
1. Created in `unresolved/` at session end
2. Processed by `session-resolver` skill
3. Moved to `resolved/` after processing

**Special rules:**
- Never manually edit resolved session logs
- Never delete session logs
- Never create session logs manually

### Getting Started Guides

**Target directory:** `docs/getting-started/`

**When to use:**
- Onboarding documentation
- Setup instructions
- Usage guides
- Deployment guides

**Naming convention:**
- Use kebab-case
- Be specific: `local-setup.md`, `env-setup.md`, `testing-guide.md`
- Avoid generic names: not `guide.md` or `howto.md`

**Required metadata:**
- `doc_id`: `GS-NNN` (e.g., `GS-001`)
- All standard metadata fields

### Templates

**Target directory:** `docs/templates/`

**When to use:**
- Creating new documentation templates
- Updating existing templates

**Naming convention:**
- Format: `[type]-template.md`
- Example: `adr-template.md`, `test-report-template.md`
- Always end with `-template.md`

**Special rules:**
- Templates define structure for other docs
- Changes to templates affect all future docs of that type
- Never modify templates without considering impact on existing docs

## File Naming Conventions

### General Rules

- **Always use kebab-case**: `my-document.md`, not `MyDocument.md` or `my_document.md`
- **Be descriptive**: `invoice-module.md`, not `module.md`
- **Avoid abbreviations**: `architecture` not `arch`, unless it's a standard prefix like `DEC-`
- **Use singular nouns**: `invoice-module.md`, not `invoices-modules.md`
- **No spaces**: use hyphens, not spaces or underscores

### Numbered Documents

For documents with sequential numbering (ADRs, reports):
- **Format**: `[PREFIX]-[NNN]-[description].md`
- **Prefix**: `DEC`, `REP-TEST`, `REP-REFACTOR`, `REP-COMMIT`
- **Number**: Three-digit zero-padded (001, 042, 123)
- **Description**: Brief kebab-case description

**Examples:**
- `DEC-003-testing-stack.md`
- `REP-TEST-001-schema-harness-smoke.md`
- `REP-REFACTOR-002-agent-config-update.md`

### Determining Next Number

Before creating a numbered document:
1. List all files in the target directory
2. Find the highest existing number for that prefix
3. Increment by one
4. Zero-pad to three digits

**Example:**
```
Existing: DEC-001, DEC-002, DEC-005
Next: DEC-006 (not DEC-003, even though it's missing)
```

## Special Files

### AGENTS.md

**Location:** Repository root

**Purpose:** Project state, agent roster, phase log, hand-off notes

**Modification rules:**
- Only modify when explicitly requested
- Never delete historical entries
- Maintain phase log table format
- Keep hand-off notes chronological

### README.md

**Location:** Repository root

**Purpose:** Project overview, public-facing documentation

**Modification rules:**
- Only modify when explicitly requested
- Maintain existing narrative structure
- Keep tone welcoming but professional
- Update version numbers when content changes

## Placement Decision Tree

When you need to write documentation, follow this decision tree:

1. **Is it about a specific code change?**
   - Refactor → `docs/reports/refactor-reports/`
   - Commit → `docs/reports/commit-reports/`
   - Test results → `docs/reports/test-reports/`

2. **Is it about a decision?**
   - Architectural decision → `docs/decisions/`
   - Technology choice → `docs/decisions/`

3. **Is it about system design?**
   - Module architecture → `docs/architecture/`
   - System overview → `docs/architecture/`
   - Data flow → `docs/architecture/`

4. **Is it about planning?**
   - Task breakdown → `docs/plans/`
   - Roadmap → `docs/plans/`
   - Implementation strategy → `docs/plans/`

5. **Is it about onboarding or usage?**
   - Setup guide → `docs/getting-started/`
   - Usage guide → `docs/getting-started/`
   - Deployment guide → `docs/getting-started/`

6. **Is it a template?**
   - New template → `docs/templates/`

7. **Is it AGENTS.md or README.md?**
   - Update in place at repository root

8. **None of the above?**
   - Flag as unclear and ask for clarification
   - Do not guess at placement

## When Placement is Unclear

If the document type doesn't clearly map to a directory:

1. **Stop** — do not guess
2. **Output:**
   ```
   NEED MORE INFORMATION
   Document type unclear. The content could fit in [option 1] or [option 2].
   Please specify target directory or clarify document type.
   ```
3. **Wait for clarification** — do not proceed with arbitrary placement
